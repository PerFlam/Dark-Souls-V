## 一、作业题目

Pulse Harbor：基于 C++ 与 Win32 图形界面的陨石生存躲避小游戏

## 二、开发软件

- 编程语言：C++
- 开发环境：Windows
- 编译器：g++ 14.2.0（MSYS2 UCRT64）
- 图形化平台：Win32 API + GDI
- 版本管理平台：GitHub

## 三、课题要求

本项目按照课程大作业要求，使用 C++ 完成一个图形化小程序。程序采用原生 Windows 图形界面，不依赖第三方游戏引擎或图形库，能够独立运行。项目内容为一款带有实时交互、碰撞检测、状态切换、难度选择和分数统计功能的小游戏，兼顾完整性、可玩性与一定的创意性。

本项目的主要目标如下：

- 设计并实现一个可以实际游玩的图形化小游戏。
- 使用面向对象思想组织核心逻辑。
- 实现开始界面、游戏进行、暂停界面、结束界面等完整流程。
- 实现碰撞检测、动态难度、模式切换、计分机制等核心功能。
- 为后续项目展示视频和实验报告提供较清晰的技术路线。

## 四、主要流程

### 1. 项目整体思路

本项目实现的是一款“生存躲避”类小游戏。玩家控制一个飞船在窗口中移动，躲避不断靠近的陨石。玩家还可以消耗能量释放脉冲技能，清除周围一定范围内的陨石。被清除的陨石有概率掉落绿色能量碎片，玩家拾取后可以补充能量，从而形成“躲避危险、积攒能量、释放技能、继续生存”的循环。

为了提升完成度与趣味性，本版本在原有基础上增加了“普通模式”和“困难模式”。玩家在开始界面即可选择不同模式，其中困难模式会减少生命值、提高陨石速度、缩短出生间隔、降低资源恢复效率，同时给予更高得分倍率，使得游戏既有挑战性也有重复游玩的价值。

### 2. 程序结构设计

程序采用面向对象的方式组织，核心类为 `Game` 类。该类统一管理游戏状态、输入处理、实体更新、碰撞检测、界面绘制和场景切换。

主要数据结构如下：

- `Vec2`：二维向量结构，用于表示位置、速度等信息。
- `Player`：保存玩家位置、生命值、速度、能量和无敌时间等状态。
- `Meteor`：保存陨石的位置、速度、半径和运动扰动参数。
- `EnergyShard`：表示能量碎片，用于能量恢复与得分奖励。
- `PulseRing`：表示释放脉冲技能时的环形特效。
- `Star`：用于背景星点绘制，提升界面表现力。
- `Scene`：枚举类型，用于表示菜单、游戏中、暂停、结束等不同状态。
- `DifficultyMode`：枚举类型，用于表示普通模式与困难模式。
- `ModeConfig`：保存不同模式下的生命值、陨石生成速度、技能消耗、得分倍率等配置。

程序入口为 `WinMain`，通过创建窗口、注册消息回调函数、启动消息循环，使整个游戏在 Windows 桌面环境下运行。

### 3. 游戏流程

游戏运行流程如下：

1. 程序启动后进入主菜单界面。
2. 玩家通过左右方向键或数字键 `1`、`2` 选择模式。
3. 玩家按下 `Enter` 键后开始游戏。
4. 游戏中通过 `WASD` 或方向键控制角色移动。
5. 系统持续生成陨石，并依据时间动态提高威胁等级。
6. 玩家碰到陨石会失去生命值，并获得短暂无敌时间。
7. 玩家按下 `Space` 可释放脉冲，消耗能量并清除周围陨石。
8. 消灭陨石后有概率产生能量碎片，拾取后恢复能量。
9. 当生命值降为 0 时，进入结算界面。
10. 玩家可按 `R` 或 `Enter` 以当前模式重新开始，也可以按 `M` 返回菜单重新选择模式。

### 4. 核心算法与实现

#### （1）游戏循环

本项目采用基于 Windows 定时器的刷新机制，通过 `SetTimer` 每约 16ms 触发一次更新，形成接近 60 帧的运行效果。每次刷新时，程序会计算与上一帧之间的时间间隔 `dt`，再根据 `dt` 更新玩家、陨石、碎片和特效等对象状态。

这种做法结构清晰，适合课程项目实现，同时能够保证游戏速度与机器性能之间相对独立，避免不同设备运行速度差异过大。

#### （2）输入处理

程序通过 `WM_KEYDOWN` 和 `WM_KEYUP` 消息记录按键状态，实现连续移动和技能触发。记录按键状态比单次触发更适合实时交互类游戏。

本项目的输入设计如下：

- `WASD` / 方向键：控制移动方向。
- `Shift`：进入精细移动状态。
- `Space`：释放脉冲技能。
- `Esc` / `P`：暂停与继续游戏。
- `1` / `2` 或左右方向键：菜单中切换模式。
- `M`：结算界面返回菜单。

#### （3）碰撞检测

由于游戏中的主要对象都可近似看作圆形，因此采用圆形碰撞检测。设两个圆心分别为 `A` 和 `B`，半径分别为 `r1` 和 `r2`，若两圆心距离小于等于半径之和，则视为发生碰撞。

判断公式为：

`(A.x - B.x)^2 + (A.y - B.y)^2 <= (r1 + r2)^2`

该方法计算量小，适合实时游戏场景。

#### （4）动态难度提升

为了避免游戏一开始过难或后期过于单调，程序中将威胁等级 `difficulty` 与生存时间关联。随着时间增加：

- 陨石生成间隔逐渐缩短。
- 陨石半径上限逐渐提高。
- 陨石速度整体增加。

这样可以形成较自然的难度曲线，使玩家在前期适应操作，在后期体验更强的挑战。

#### （5）困难模式设计

困难模式不是简单地提高一个数字，而是从多个维度同时调整游戏节奏。程序通过 `ModeConfig` 结构体集中管理不同模式参数，游戏启动时根据当前模式读取对应配置。

普通模式与困难模式的差异如下：

| 项目 | 普通模式 | 困难模式 |
| --- | --- | --- |
| 初始生命值 | 3 | 2 |
| 初始陨石生成等待 | 0.45 秒 | 0.30 秒 |
| 最短生成间隔 | 0.22 秒 | 0.14 秒 |
| 难度增长速度 | 标准 | 更快 |
| 陨石速度 | 标准 | 更快 |
| 能量自然恢复 | 较快 | 较慢 |
| 脉冲消耗 | 35 | 42 |
| 脉冲冷却 | 0.75 秒 | 0.95 秒 |
| 掉落概率 | 较高 | 较低 |
| 得分倍率 | 1.0 倍 | 1.55 倍 |

这样设计的优点是：困难模式在玩法上能给玩家明显压力，但又通过更高得分倍率保留正向反馈，使其更适合展示“不同参数设计带来的体验差异”。

#### （6）能量与技能机制

玩家拥有能量条，能量会随时间缓慢恢复，也可以通过拾取能量碎片加快恢复。当能量足够且技能不在冷却中时，玩家可以释放脉冲技能。

脉冲技能的作用包括：

- 清除一定范围内的陨石。
- 形成环形视觉特效。
- 提升游戏节奏感和操作反馈。

在普通模式中，技能偏向“容错”；而在困难模式中，技能更像“关键资源”，需要更慎重地把握释放时机。

#### （7）双缓冲绘制

为减少窗口绘制闪烁，程序在绘图时先创建内存设备上下文，将背景、角色、陨石、UI 等内容绘制到内存位图，再一次性拷贝到窗口。这种方式能够显著提升画面稳定性，使运行效果更平滑。

### 5. 图形界面设计

虽然本项目未使用外部美术资源，但仍通过颜色和布局设计增强界面表现：

- 深色背景配合动态星点，营造太空环境氛围。
- 玩家采用亮蓝色圆形与外圈描边，便于在场景中识别。
- 陨石使用橙黄色配色，与玩家形成明显对比。
- 绿色能量碎片用于提示可收集资源。
- 顶部区域显示得分、生存时间、威胁等级、当前模式、能量条和技能冷却条。
- 开始界面新增模式选择说明与两种模式的最佳成绩展示。
- 中央弹窗用于显示开始、暂停和结束界面信息。

这种设计虽然实现方式较基础，但能够保证界面清晰、功能完整，并具有一定视觉效果。

### 2. 普通模式与困难模式对比测试

为了验证困难模式确实对游戏体验产生影响，进行了人工对比测试，结果如下：

| 对比项目 | 普通模式表现 | 困难模式表现 |
| --- | --- | --- |
| 前 10 秒压力 | 适中 | 明显更高 |
| 脉冲释放频率 | 较容易积攒 | 需要更谨慎使用 |
| 容错空间 | 较大 | 较小 |
| 生存时间上限 | 更容易提升 | 更依赖熟练操作 |
| 得分增长 | 平稳 | 风险更高但得分倍率更高 |

测试结果表明，困难模式的节奏、压力和容错空间都与普通模式有明显区分，达到了“模式切换带来真实玩法变化”的设计目标。

### 3. 测试结果分析

当前版本中，普通模式和困难模式均已完成并可运行。编译后能够成功生成可执行文件，程序可在本地正常打开并游玩。测试过程中重点关注了以下问题：

- 玩家是否会移出窗口边界。

- 困难模式下的技能冷却、得分倍率和掉落概率是否生效。
- 陨石与玩家碰撞后是否正确扣血。
- 暂停后时间和实体运动是否被正确停止。
- 结束界面是否能够正常重开游戏或返回菜单。

经过多次运行测试，上述逻辑均能正常工作。

## 六、项目特色与创新点

相较于普通的“鼠标点击”或“简单绘图”类图形程序，本项目的创新点主要体现在以下几个方面：

- 不是静态演示程序，而是具有完整交互流程的实时小游戏。
- 在生存躲避基础上加入了能量资源管理与范围技能机制。
- 采用动态难度提升，使游戏过程具有明显节奏变化。
- 增加了普通模式与困难模式，使同一核心玩法形成不同体验。
- 使用纯 Win32 API 与 GDI 完成界面渲染，减少第三方依赖，便于理解底层图形化程序结构。
- 程序包含菜单、暂停、结算、重开、记分、模式选择等较完整的产品化元素，更适合作为课程展示作品。

## 七、收获与体会

通过完成本项目，我对 C++ 图形化程序开发和游戏逻辑组织有了更直观的理解。

首先，我进一步熟悉了 Windows 消息机制。与控制台程序不同，图形界面程序需要围绕消息循环组织输入与界面刷新，这对程序结构设计提出了更高要求。

其次，我体会到了面向对象设计在中型程序中的作用。若将所有功能都写在主函数中，代码会非常混乱；而通过 `Game` 类集中管理状态，再将玩家、陨石、碎片、模式配置等对象分别建模，程序结构会更加清晰，也更便于后续扩展。

再次，我学习了图形化程序中的基础实时算法，包括向量计算、碰撞检测、定时刷新、双缓冲绘图等。这些内容虽然公式不复杂，但在实际项目中组合起来后，能够构成一个完整可玩的系统。

本次增加困难模式后，我也进一步理解了“游戏参数设计”的意义。同样的程序逻辑，通过生命值、生成速度、资源恢复和计分倍率等参数的变化，就能让玩家获得差异明显的体验。这说明程序设计不仅包括代码实现，也包括规则设计与体验调优。
