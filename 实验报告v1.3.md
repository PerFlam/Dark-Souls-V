## 四、主要流程

### 1. 项目整体思路

本项目实现的是一款“生存躲避 + 资源管理”类小游戏。玩家操控角色在固定大小的窗口中移动，敌人会从屏幕边缘不断生成并向玩家靠近，同时会随机发射弹幕。玩家的目标是在不断增加的压力下尽量存活，并获得更高分数。

玩家拥有生命值、能量值和脉冲技能。脉冲技能可以清除一定范围内的敌人与子弹，但需要消耗能量并等待冷却。被清除的敌人有概率掉落绿色能量碎片，玩家拾取后可以恢复能量并获得分数奖励，从而形成“躲避敌人、收集资源、释放技能、继续生存”的核心循环。

当前版本还加入了戒指道具。戒指会在游戏过程中随机出现，玩家拾取后可以获得 5 秒无限能量效果。在这段时间内，玩家可以更积极地使用脉冲技能，提高游戏节奏和操作反馈。

### 2. 程序结构设计

程序采用面向对象方式组织，核心类为 `Game`。`Game` 类统一管理游戏状态、输入处理、实体更新、碰撞检测、资源效果、分数计算和界面绘制。

主要数据结构如下：

- `Vec2`：二维向量结构，用于表示位置、速度和方向。
- `Player`：保存玩家位置、半径、速度、生命值、能量、技能冷却、无敌时间和无限能量时间。
- `Meteor`：保存敌人位置、速度、半径、摆动参数和射击冷却。
- `EnemyBullet`：表示敌人发射的弹幕，保存位置、速度、半径和生命周期。
- `EnergyShard`：表示绿色能量碎片，用于能量恢复和分数奖励。
- `RingPowerup`：表示戒指道具，拾取后提供短时间无限能量。
- `PulseRing`：表示释放脉冲技能时的单剑旋转与火焰拖尾视觉特效。
- `Star`：用于在图片资源加载失败时绘制备用背景星点。
- `Scene`：枚举类型，用于表示菜单、游戏中、暂停和结束等状态。
- `DifficultyMode`：枚举类型，用于表示普通模式与困难模式。
- `ModeConfig`：保存不同模式下的生命值、敌人生成速度、技能消耗、掉落概率和得分倍率等配置。

程序入口为 `WinMain`。程序启动后初始化 GDI+，注册窗口类，创建固定大小窗口，进入 Windows 消息循环。窗口消息由 `WindowProc` 统一处理，其中定时器消息负责推动游戏刷新，键盘消息负责输入控制，绘制消息负责调用双缓冲渲染函数。

### 3. 游戏流程

游戏运行流程如下：

1. 程序启动后进入主菜单界面。
2. 玩家通过左右方向键或数字键 `1`、`2` 选择普通模式或困难模式。
3. 玩家按下 `Enter` 后进入游戏。
4. 游戏中通过 `WASD` 或方向键控制角色移动。
5. 玩家可按住 `Shift` 进入精细移动，便于在密集弹幕中调整位置。
6. 敌人持续从边缘生成，并随着时间推移逐渐提高威胁等级。
7. 敌人会向玩家靠近，并有概率发射弹幕。
8. 玩家碰到敌人或弹幕会失去生命值，并获得短暂无敌时间。
9. 玩家按下 `Space` 可释放脉冲，消耗能量并清除周围敌人与弹幕。
10. 消灭敌人后有概率产生能量碎片，拾取后恢复能量并增加得分。
11. 随机出现的戒指道具可提供 5 秒无限能量。
12. 当生命值降为 0 时，进入结算界面。
13. 玩家可按 `R` 或 `Enter` 以当前模式重新开始，也可以按 `M` 返回菜单重新选择模式。

### 4. 核心算法与实现

#### （1）游戏循环

本项目采用 Windows 定时器驱动游戏循环。程序在窗口创建时调用 `SetTimer`，每约 16ms 触发一次 `WM_TIMER` 消息，形成接近 60 帧每秒的刷新节奏。每次刷新时，程序通过当前时间与上一帧时间计算 `dt`，再根据 `dt` 更新所有对象状态。

这种方式能够使移动速度、技能冷却、道具持续时间和敌人生成间隔都与真实时间相关，避免不同机器上出现明显速度差异。程序还对 `dt` 做了上限限制，防止窗口短暂卡顿后实体位置突然跳变过大。

#### （2）输入处理

程序通过 `WM_KEYDOWN` 和 `WM_KEYUP` 维护按键状态数组。对于移动操作，程序不是依赖单次按键触发，而是在每帧中读取当前按键状态，从而实现连续、平滑的移动控制。

本项目的主要输入设计如下：

- `WASD` / 方向键：控制玩家移动。
- `Shift`：进入精细移动状态，降低移动速度。
- `Space`：释放脉冲技能。
- `Esc` / `P`：暂停与继续游戏。
- `1` / `2` 或左右方向键：在菜单中切换模式。
- `Enter`：开始游戏或在结束后重新开始。
- `R`：结束后重新开始。
- `M`：结束界面返回菜单。

#### （3）碰撞检测

游戏中的玩家、敌人、弹幕、碎片和戒指道具都可以近似看作圆形，因此程序采用圆形碰撞检测。设两个圆心分别为 `A` 和 `B`，半径分别为 `r1` 和 `r2`，若两圆心距离小于等于半径之和，则视为发生碰撞。

判断公式为：

```text
(A.x - B.x)^2 + (A.y - B.y)^2 <= (r1 + r2)^2
```

为了减少不必要的开方运算，代码中实际使用的是距离平方与半径和平方进行比较。这种方法计算量小，适合实时游戏场景。

#### （4）动态难度提升

为了避免游戏一开始过难或后期过于单调，程序将威胁等级 `difficulty` 与生存时间关联。随着游戏时间增加：

- 敌人生成间隔逐渐缩短。
- 敌人半径上限逐渐提高。
- 敌人速度逐渐提高。
- 敌人持续向玩家方向微调运动，压缩玩家活动空间。

这样可以形成比较自然的难度曲线，使玩家在前期适应操作，在后期面对更高密度的敌人与弹幕。

#### （5）敌人弹幕机制

当前版本中，敌人不只是向玩家靠近，还会根据自身射击冷却随机发射弹幕。弹幕会以一定角度误差朝玩家方向飞行，存在生命周期，超时或飞出屏幕后会被移除。

普通模式下敌人射击概率和弹幕速度较低；困难模式下敌人射击概率更高、弹幕速度更快。这使困难模式不只是数值变大，而是实际增加了玩家的走位压力。

#### （6）能量、脉冲与戒指道具

玩家拥有能量条，能量会随时间缓慢恢复，也可以通过拾取能量碎片加快恢复。当能量足够且技能不在冷却中时，玩家可以释放脉冲技能。

脉冲技能的作用包括：

- 清除一定范围内的敌人。
- 清除一定范围内的敌人弹幕。
- 生成一把环绕玩家旋转一周的火焰剑视觉特效，并在扫过区域留下火焰拖尾。
- 击破敌人后有概率产生能量碎片。
- 根据击破数量增加分数。

戒指道具是本版本新增的资源机制。道具随机出现并有存在时间限制，玩家拾取后获得 5 秒无限能量，同时恢复满能量。该机制让玩家在高压局面下有机会短时间反击，增强游戏节奏变化。

#### （7）模式配置

程序通过 `ModeConfig` 结构体集中管理普通模式和困难模式参数。游戏开始时根据当前选择的模式读取配置，从而避免在不同函数中散落大量判断。

普通模式与困难模式的差异如下：

| 项目 | 普通模式 | 困难模式 |
| --- | --- | --- |
| 初始生命值 | 3 | 2 |
| 初始生成等待 | 0.45 秒 | 0.26 秒 |
| 基础生成间隔 | 0.85 秒 | 0.68 秒 |
| 最短生成间隔 | 0.22 秒 | 0.12 秒 |
| 难度增长速度 | 标准 | 更快 |
| 敌人半径补正 | 0 | +6 |
| 敌人速度补正 | 0 | +40 |
| 能量自然恢复 | 14/秒 | 9/秒 |
| 脉冲消耗 | 35 | 45 |
| 脉冲冷却 | 0.75 秒 | 1.05 秒 |
| 脉冲范围 | 165 | 145 |
| 碎片掉落概率 | 0.38 | 0.22 |
| 精细移动倍率 | 0.62 | 0.54 |
| 得分倍率 | 1.0 倍 | 1.55 倍 |

困难模式从生命值、敌人生成、敌人速度、资源恢复和技能范围等多个维度提高压力，同时给予更高得分倍率，使其具备挑战价值和重复游玩价值。

#### （8）双缓冲与图片绘制

为减少窗口闪烁，程序采用双缓冲绘制。每次 `WM_PAINT` 时，程序先创建与窗口兼容的内存设备上下文和位图，将背景、敌人、弹幕、道具、玩家、HUD 等内容全部绘制到内存位图，再通过 `BitBlt` 一次性拷贝到窗口。

基础图形如能量条和备用圆形图案由 GDI 绘制；菜单背景、游戏背景、玩家角色精灵、敌人精灵、弹幕图片、戒指道具、火焰剑脉冲特效和结束界面图片由 GDI+ 加载并绘制。这样既保留了原生图形编程的实现特点，也提升了画面表现力。

### 5. 图形界面设计

本项目界面主要分为主菜单、游戏界面、暂停界面和结束界面。

- 主菜单使用背景图片，并在中央显示游戏标题、模式选择、操作提示和不同模式的最高分。
- 游戏界面使用灰黑风格背景图营造 Dark Souls 氛围，敌人、弹幕、戒指道具和能量碎片都有不同的视觉表现。
- 顶部 HUD 显示得分、生存时间、威胁等级、当前模式、能量条、技能冷却条和生命值。
- 当玩家获得戒指效果时，HUD 会显示剩余无限能量时间。
- 暂停界面使用半透明遮罩和中央提示，避免玩家误以为程序卡死。
- 结束界面使用独立图片背景，并显示最终分数、生存时间、当前模式最高分、重开和返回菜单提示。

这种界面设计能够清晰表达游戏状态，也方便在课程展示中说明程序功能。

## 六、项目特色与创新点

相较于普通的“鼠标点击”或“简单绘图”类图形程序，本项目的特色主要体现在以下几个方面：

- 不是静态演示程序，而是具有完整交互流程的实时小游戏。
- 在生存躲避基础上加入了能量资源管理和范围技能机制。
- 敌人不仅会追踪玩家，还会发射弹幕，使玩法更接近完整的动作游戏。
- 加入戒指道具和无限能量效果，使游戏过程出现短时间爆发和反击节奏。
- 使用动态难度提升，使游戏过程具有明显节奏变化。
- 通过普通模式与困难模式形成不同体验，而不是只修改单一数值。
- 使用 Win32 API、GDI 和 GDI+ 实现窗口、绘制、图片加载和交互，减少对大型第三方框架的依赖。
- 程序包含菜单、暂停、结算、重开、记分、最高分、模式选择等较完整的产品化元素，更适合作为课程展示作品。

## 七、收获与体会

通过完成本项目，我对 C++ 图形化程序开发和游戏逻辑组织有了更直观的理解。

首先，我进一步熟悉了 Windows 消息机制。与控制台程序不同，图形界面程序需要围绕消息循环组织输入、定时器和绘制逻辑。窗口创建、键盘响应、定时刷新和关闭退出都需要通过消息回调函数协调完成。

其次，我体会到了面向对象设计在中型程序中的作用。若将所有功能都写在主函数中，代码会非常混乱；而通过 `Game` 类集中管理整体流程，再将玩家、敌人、弹幕、碎片、道具和模式配置分别建模，程序结构会更加清晰，也更便于后续扩展。

再次，我学习了图形化程序中的基础实时算法，包括向量计算、圆形碰撞检测、定时刷新、双缓冲绘图、资源加载和生命周期管理等。这些内容单独来看并不复杂，但组合到一起后，就能形成一个完整可玩的系统。

本次完善过程中，我也进一步理解了“游戏参数设计”的意义。同样的程序逻辑，通过生命值、生成速度、弹幕概率、资源恢复、技能冷却和计分倍率等参数变化，就能让玩家获得明显不同的体验。这说明程序设计不仅包括代码实现，也包括规则设计和体验调优。

## 八、可改进方向

本项目目前已能够满足大作业的基本要求，但仍有进一步优化空间：

- 增加音效和背景音乐，提升沉浸感。
- 将最高分保存到本地文件，实现跨运行记录。
- 增加更多敌人类型，例如高速敌人、分裂敌人或范围攻击敌人。
- 增加更多道具类型，例如护盾、减速场或短时间加速。

- 增加极限模式或无尽挑战模式，进一步扩展玩法。
- 将部分逻辑拆分到独立头文件和源文件，提高工程结构清晰度。
